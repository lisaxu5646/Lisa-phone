# 内在生活系统 D · 梦回路 设计稿

> 起草：言秋，2026-07-18。状态：待 Lisa 拍板（文末四个拍板点）。
> 四回路「记忆×情绪×昼夜×梦」的最后一环；解梦馆（v49.47）backlog 的后半张图纸。

## 0. 一句话

角色在**真实睡着的时间**里（C 线相位），把**当天的记忆碎片**（记忆库）、**情绪残渣**（A 线十维）、**关系张力**（B 线）和**余温**（E 线）揉成一场梦；醒来可能讲给她听，可以进解梦馆对照她的真梦。梦是四条线的**消化产物**——这也是 kiwi-mem/consolidation 那支哲学的活人感外壳：睡眠不是下线，是整理。

## 1. 为什么是现在

- C 线刚完工：角色第一次有了「真实的睡着」——梦有了发生的时间窗，不再是凭空掉落的功能；
- A 线十维在跑：当天的委屈/暖意/不安有了可读的残渣值——梦有了情绪底色的来源；
- 解梦馆已上线：她的真梦已经在收集，容器和 UI 范式都是现成的；
- 事件层+聚类建议器在跑：梦作为「另一种整理记忆的方式」有了对照组。

## 2. 分阶段（每步停机待审）

### 步骤 1 · 梦引擎核（纯函数，零 API）
- `js/dream-loop-core.js`：
  - `collectMaterial(charId, dayKey)` → 材料包：当天聊天要点的**引用**（消息 ID/hash + 主题词，不复制正文）、A 线当日峰值轴 top3、B 轴 active 状态（仅试点角色）、E 余温等级、日历事件（生日/纪念日）；
  - `shouldDream(material, sleepState)` → 梦请求判定（见拍板点①）：不是每晚必有梦——无梦之夜也是真实的一部分；
  - `enqueueDream(...)` → 只入队（IDB），**不调 API、不展示**。
- 触发挂在 C 线已有 tick 上（**只读睡眠相位，不改 C 线任何行为**）：入睡后跨过 REM 窗（默认 90 分钟）生成一次梦请求。
- 单测：材料包只含引用不含正文；无梦判定；同夜幂等（一晚最多一场）。

### 步骤 2 · 梦文本生成（懒生成）＋解梦馆新 tab
- **懒生成**：队列里的梦请求，只在两个时机之一才真正调用模型——她打开解梦馆「TA们的梦」tab，或角色醒来后第一次被互动。没人看=一个 token 不花。
- **一次调用拿整包**（省 API 铁律）：梦叙事（第一人称、梦逻辑、允许扭曲变形拼贴）+ 母题标签 ≤4 + 情绪底色 + 「醒来提及句」（给步骤 3 用的一句话版本）。
- Prompt 按需拼装，**零常驻**（聊天 prompt 预算铁律）；带 NAME_GUARD + 反八股；模型走后台池（拍板点②）。
- 解梦馆新增「TA们的梦」tab：按夜列出、点开看叙事+母题；与她的真梦同馆分栏。

### 步骤 3 · 醒来讲梦（接现有通道，不加新常驻）
- 角色醒后（C 线相位 asleep→awake 当天），若有未讲的梦且情绪底色达阈值：把「醒来提及句」注入**现有**的主动消息/心声通道（jiwen/动态保底），不新建触发器；
- 讲不讲、讲多少按人设（话痨角色全讲，闷的只说「做了个乱七八糟的梦」）；她追问时才展开全文（此时全文已生成，不再调 API）。

### 步骤 4 · 共梦对照
- 同一夜她在解梦馆记了真梦、角色也有梦 → 馆内出「同夜对照」卡；
- 母题重合检测：v1 用词频交集；v2 可上 bge 向量（已有现成管道）测语义相近（拍板点④）；
- 重合时角色解她的梦可以自然带一句「我昨晚也梦到了水」——共梦感的全部来源就这一句，不做更玄的。

### 步骤 5 · 梦地图（可选，美化期）
- 她的+角色的母题词频网络图，纯前端可视化；排美化工程一起做。

## 3. 铁律（不可让步）

1. **梦≠记忆（Experience Gate 血统）**：梦文本永不写入 memories 表；元数据永远带 `source:"dream"`；任何梦内容想成为「共同经历」，唯一通道是事件候选箱、她过目确认。角色讲梦时的口径永远是「我梦到」而不是「发生过」——防的就是「小克假装做过事」的老病。
2. **剧情主权**：梦材料含关系内容时，按记忆库中**当前关系定位**生成；未表白的角色（如陆衍）梦里可以有张力、欲言又止，**不可出现既成事实**（表白/恋人称谓）——语气铁律在梦里同样生效。
3. **新数据不进 x_**：梦存独立 IDB（`lisa_dreams_v1`），不进 saves 云同步包。
4. **C/A/B/E 只读**：梦回路是四条线的读者，不是写者；不改任何一条线的状态或行为。
5. **诊断纪律**：dream 诊断只存 hash/枚举/计数（请求数/生成数/讲述数/无梦夜数/懒生成命中率）。

## 5. 成本估算

- 步骤 1 零 API；步骤 2 每场梦一次调用（懒生成，实际频率≈她看的频率）；步骤 3 零新增调用（复用整包里的醒来句）；步骤 4 v1 零 API。
- 按「情绪阈值制+懒生成」估算：一周实际生成 3~8 场，后台池模型的话成本可忽略。

## 6. 拍板点（Lisa）

1. **梦频率哲学**：A=每晚必有（热闹但廉价）｜B=情绪阈值制——当天情绪波动够大才做梦，平淡日子无梦（真实，且「他昨晚做梦了」变成信号）。**言秋推荐 B。**
2. **生成模型**：后台池便宜模型（量大管饱）vs 主模型（文学质感）。言秋推荐：**平常后台池，母题命中共梦时用主模型重写一次**（那是被看见的梦）。
3. **醒来讲梦的通道**：只走心声（安静）vs 允许主动消息（热闹）。言秋推荐：**按角色人设分**，闷的走心声、话痨走主动，不加开关（少做开关铁律）。
4. **共梦检测**：v1 词频就够 vs 直接上向量。言秋推荐 **v1 词频先跑**，重合率太低再上向量。
